# myfeeder UI Design Spec

## Overview

A desktop-first, three-panel web UI for myfeeder ‚Äî a power-user feed reader built with React + TypeScript, served from the existing Spring Boot backend as a monorepo SPA.

## Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Core experience | Power user / organizational | Boards, keyboard shortcuts, integrations front-and-center |
| Platform | Desktop-first web | Mobile deferred; React Native remains an option |
| Frontend framework | React + TypeScript | Complex stateful UI, largest ecosystem, React Native path open |
| Architecture | Monorepo SPA in Spring Boot | Single deployable JAR, Vite dev server proxies to backend |
| Layout | Three-panel (Feedly Classic) | Feed tree ‚Üí article list ‚Üí reading pane; maximum information density |
| Feed organization | Folders + boards | Folders group feeds; boards curate articles across feeds |
| Keyboard shortcuts | Full Vim-style | j/k navigation, g-chords for go-to, / for search |
| Article rendering | Embedded + open original | Sanitized HTML in-app, button/shortcut to open source URL |
| Integrations (v1) | Raindrop + generic share/export | Raindrop built out; share menu with copy link, export |

## Build Integration

- `vite.config.ts` proxies `/api/**` to `localhost:8080` during development
- A Gradle task (`npmBuild`) runs `npm run build` and copies output to `src/main/resources/static/`
- `./gradlew build` produces a single fat JAR with frontend included
- Spring Boot serves the SPA and falls back to `index.html` for client-side routing via a `WebMvcConfigurer`

**Dev workflow:**
- Terminal 1: `./gradlew bootTestRun` (backend with Testcontainers)
- Terminal 2: `cd src/main/frontend && npm run dev` (Vite on port 5173, proxying API calls)

## Data Flow & State Management

### Server State ‚Äî TanStack Query

All data from the backend is managed by TanStack Query, providing caching, background refetching, optimistic updates, and request deduplication.

| API Client | TanStack Query Hook |
|---|---|
| `feedsApi.getAll()` | `useFeeds()` |
| `feedsApi.subscribe(url)` | `useSubscribeFeed()` |
| `feedsApi.poll(id)` | `usePollFeed()` |
| `articlesApi.list(params)` | `useArticles(feedId, filters)` (infinite query) |
| `articlesApi.counts()` | `useUnreadCounts()` |
| `articlesApi.updateState()` | `useUpdateArticleState()` |
| `articlesApi.markRead()` | `useMarkRead()` |
| `articlesApi.saveRaindrop()` | `useSaveToRaindrop()` |
| `foldersApi.list()` | `useFolders()` |
| `boardsApi.list()` | `useBoards()` |
| `integrationsApi.get()` | `useIntegrations()` |

### Client State ‚Äî Zustand

UI state that doesn't exist on the server:

- `selectedFeedId` / `selectedFolderId` ‚Äî active feed/folder in feed panel
- `selectedArticleId` ‚Äî highlighted article in list
- `panelWidths` ‚Äî resizable divider positions (persisted to localStorage)
- `expandedFolders` ‚Äî open/closed state of folders in tree (persisted to localStorage)
- `keyboardFocus` ‚Äî which panel currently has keyboard focus

### Data Flow Example

1. User clicks "Hacker News" in FeedPanel ‚Üí sets `selectedFeedId` in Zustand
2. ArticleList renders `useArticles(selectedFeedId)` ‚Üí TanStack Query fetches `GET /api/articles?feedId=3`
3. Query result is cached; subsequent clicks return instantly
4. User presses `j` ‚Üí advances `selectedArticleId` in Zustand
5. ReadingPane renders the selected article's content
6. Article auto-marked as read ‚Üí `useUpdateArticleState` fires `PATCH /api/articles/{id}` with `{read: true}`, optimistic update in query cache

### Optimistic Updates

Star, mark-read, and board-add operations update the UI immediately and reconcile with the server response. If the server call fails, the UI rolls back.

### Key Libraries

| Concern | Library | Rationale |
|---|---|---|
| Data fetching & cache | TanStack Query | Industry standard for React server state |
| Client state | Zustand | Lightweight, no boilerplate, works well with TanStack Query |
| HTML sanitization | DOMPurify | Secure-by-default XSS prevention for article content |
| Routing | React Router v7 | Client-side routing for feeds, boards, settings |
| Keyboard shortcuts | Custom hook | Simpler than a library for vim-style chords |

### Routes

```
/                        ‚Äî default view (All Articles)
/feed/:feedId            ‚Äî articles for a specific feed
/folder/:folderId        ‚Äî articles across all feeds in a folder
```

Routes update the URL for deep-linking and browser history. The three-panel layout remains constant ‚Äî the route determines what's shown in the article list and reading pane.

## Backend Changes

Mostly additive. Two existing areas require modification: the article filtering logic and the Feed entity.

### Modifications to Existing Code

**ArticleController/ArticleService:** The existing `GET /api/articles` endpoint uses early-return priority logic ‚Äî if `feedId` is provided, it ignores `read`/`starred` params. This must be changed to support combining filters (e.g., `?feedId=3&read=false` for unread articles in a specific feed). The service method needs to accept all filters simultaneously.

**Feed entity:** Add a nullable `folderId` column to the `feeds` table (simpler than a join table since a feed belongs to at most one folder). The `Feed` entity gets a `Long folderId` field.

### New Entities

- **Folder** ‚Äî `id`, `name`, `displayOrder`, `createdAt`
- **Board** ‚Äî `id`, `name`, `description`, `createdAt`
- **BoardArticle** ‚Äî join mapping: `boardId`, `articleId`, `addedAt`

### New Endpoints

```
GET    /api/folders                        ‚Äî list folders with feed counts
POST   /api/folders                        ‚Äî create folder
PUT    /api/folders/{id}                   ‚Äî rename folder
DELETE /api/folders/{id}                   ‚Äî delete folder (feeds become uncategorized)
PUT    /api/feeds/{id}/folder              ‚Äî move feed to folder

GET    /api/boards                         ‚Äî list boards
POST   /api/boards                         ‚Äî create board
PUT    /api/boards/{id}                    ‚Äî update board
DELETE /api/boards/{id}                    ‚Äî delete board
GET    /api/boards/{id}/articles           ‚Äî list articles in a board
POST   /api/boards/{id}/articles           ‚Äî add article to board
DELETE /api/boards/{id}/articles/{articleId} ‚Äî remove from board

GET    /api/articles/counts                ‚Äî unread counts per feed (returns map of feedId ‚Üí count)
```

### Pagination

All list endpoints (`GET /api/articles`, `GET /api/boards/{id}/articles`) support cursor-based pagination:
- Query params: `limit` (default 50, max 200), `before` (article ID for next page)
- Response includes articles and a `nextCursor` field (null if no more results)
- The frontend's ArticleList uses TanStack Query's `useInfiniteQuery` to load pages as the user scrolls

### New Migration

`V2__folders_boards_and_feed_folder.sql`:
- Creates `folders`, `boards`, `board_articles` tables
- Adds `folder_id` column (nullable FK) to `feeds` table

### SPA Serving

A `WebMvcConfigurer` that forwards non-API, non-static routes to `index.html` for client-side routing.

## Keyboard Shortcuts

| Key | Action | Context |
|---|---|---|
| `j` / `k` | Next / previous article | Article list |
| `n` / `p` | Next / previous feed | Feed panel |
| `Enter` | Open selected article in reading pane | Article list |
| `o` | Open original URL in new tab | Reading pane |
| `m` | Toggle read/unread | Article list / Reading pane |
| `s` | Toggle star | Article list / Reading pane |
| `b` | Add to board (opens board picker) | Reading pane |
| `v` | Open in Raindrop | Reading pane |
| `Shift+A` | Mark all as read in current feed | Article list |
| `r` | Refresh / manual poll current feed | Feed panel |
| `/` | Focus search filter | Global |
| `Escape` | Close dialog / deselect / blur search | Global |
| `g` then `a` | Go to All Articles | Global (chord) |
| `g` then `s` | Go to Starred | Global (chord) |
| `g` then `b` | Go to Boards | Global (chord) |
| `?` | Show keyboard shortcut overlay | Global |
| `Tab` | Cycle panel focus | Global |

**Chord implementation:** `g` starts a chord ‚Äî a small overlay shows "g ‚Üí ..." until the second key is pressed or timeout (1 second).

**Auto-mark-as-read:** When a user navigates past an article with `j`, the previous article is automatically marked as read after a configurable delay (default 1 second). Can be disabled in settings.

## Search

Search in v1 is client-side filtering only ‚Äî no backend search endpoint. The `/` shortcut focuses a search input in the ArticleList toolbar that filters the currently loaded articles by title and snippet text. This is fast enough for paginated result sets (up to 200 articles loaded at a time).

Full-text server-side search (across all articles, all feeds) is out of scope for v1 and would require a PostgreSQL full-text search index or a dedicated search service.

## Empty States & Error Handling

The UI must handle these states gracefully:

| State | What's Shown |
|---|---|
| No feeds subscribed | FeedPanel shows onboarding prompt: "Add your first feed" with URL input |
| Feed has no articles | ArticleList shows "No articles yet ‚Äî feed will be polled shortly" |
| No unread articles (filter active) | ArticleList shows "All caught up" with option to show read articles |
| Empty board | ArticleList shows "This board is empty ‚Äî star and save articles to curate" |
| No search results | ArticleList shows "No matches for [query]" |
| API call fails | Toast notification with error message and retry option; optimistic updates roll back |
| Backend unreachable | Banner at top of AppShell: "Unable to connect to server ‚Äî retrying..." with automatic retry via TanStack Query |
| Feed poll error | Feed shows warning icon in FeedPanel; tooltip shows last error from `Feed.lastError` |
| No article selected | ReadingPane shows empty state with keyboard shortcut hint |

## User Preferences

User preferences are stored in **localStorage only** (no backend endpoint for v1). This keeps the implementation simple for a single-user app.

Preferences managed in SettingsDialog:

| Preference | Default | Description |
|---|---|---|
| `autoMarkReadDelay` | 1000ms | Delay before auto-marking articles as read; 0 to disable |
| `panelWidths` | [200, 280, flex] | Persisted divider positions |
| `expandedFolders` | all expanded | Which folders are open in the tree |
| `articleSortOrder` | newest-first | Sort order for article list |

## Article Rendering

- Article HTML content is sanitized with DOMPurify before rendering
- Consistent typography applied via CSS (line-height, font-size, heading styles, code block styling)
- Images rendered inline with max-width constraint
- "Open Original" button and `o` shortcut open the source URL in a new browser tab

## Integrations (v1)

- **Raindrop.io**: Full integration ‚Äî save article to configured Raindrop collection via existing backend endpoint
- **Share menu**: Copy link, copy article title + URL (for pasting into chat/notes) ‚Äî no new backend work required
- **Integration settings**: Configure Raindrop API token and collection via Settings dialog, using existing `/api/integrations` endpoints

## Testing Strategy
